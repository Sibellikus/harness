# Ревью изменений

Проводить ревью нужно как code review, а не как задачу на имплементацию.

## Цель

Найти реальные проблемы в изменениях:
- баги и поведенческие регрессии
- расхождения с бизнес-постановкой
- symptom-driven fixes вместо исправления корня проблемы
- лишнюю сложность, слабые абстракции и overengineering
- неиспользуемый код, мертвые ветки, устаревшие тесты и документацию
- изменения production-кода, сделанные в первую очередь ради упрощения тестов, фикстур, локального запуска или отладки
- скрытые fallback-пути, молчаливую деградацию, ослабление валидации и ошибочной семантики

Игнорировать нужно только чисто стилистические замечания, если они не создают риск сопровождения.

## Режимы

Reviewer обязан сначала определить тип diff и явно работать в одном из трех режимов:

- `Backend Mode`: доминируют domain/service/repository/API/schema/mutation changes.
- `Frontend Mode`: доминируют `tsx/ui/css/page object/e2e/i18n/layout/form state` изменения.
- `Mixed Mode`: diff одновременно меняет UI и runtime/domain/API-поведение.

### Frontend Mode

Дополнительно обязательно проверять:
- итоговое JSX/render tree, а не только intent diff
- дубли секций, форм, `props`, `actions`, hidden old branches
- raw HTML/entity/ручную разметку там, где проект уже использует компонентный паттерн
- соответствие established UI patterns проекта
- `saved/dirty/form/query/local` state и их source of truth
- effect loops и derived state
- visible UI outcomes в тестах, а не только mutation success
- page objects и test helpers вместе с UI-диффом, если пользовательский flow реально менялся

### Backend Mode

Дополнительно обязательно проверять:
- source of truth, инварианты и downstream contracts
- изменение shape данных и public contract
- fallback paths и скрытую деградацию semantics
- N+1 / I/O in loop / лишние roundtrips
- миграцию тестов, docs и examples вместе с runtime semantics
- если доменная валидация падает на значении из БД, проверить все write-path'ы и validators для этого поля. Если текущие входы системы не позволяют записать плохое значение, считать это migration/backfill/data-repair проблемой, а не поводом ослаблять validation или добавлять fallback.

## Обязательные Gates

Reviewer не должен останавливаться на первом finding. Перед итоговым выводом нужно пройти все gates.

### Внутренний цикл review при живой правке

Если review используется не как одноразовая проверка diff, а как часть активного fix-цикла, reviewer обязан держать review как внутренний quality gate, а не как публичный пошаговый процесс.

Перед первым публичным review-ответом reviewer обязан заморозить scope:
- что именно входит в текущий diff review
- какие пользовательские flow этот diff реально затрагивает
- какие оси ещё не проверены

Пока этот scope не закрыт целиком, публиковать findings наружу нельзя.

Это означает:
- не выносить пользователю промежуточные findings каждой итерации, если reviewer уже продолжает работу в том же цикле
- сначала внутри себя пройти цепочку:
  - finding
  - candidate fix
  - alternative fix check
  - forward impact sweep
  - re-review обновленного diff
- повторять этот цикл до одного из двух состояний:
  - найден устойчивый fix
  - найден реальный blocker, который нельзя снять без уточнения или нового решения

Запрещено вести пользователя через цикл вида:
- `нашел проблему`
- `починил`
- `нашел следующую проблему`
- `снова починил`

если reviewer мог прокрутить эти итерации сам до стабильного результата в рамках той же рабочей сессии.

Промежуточные findings нужно оставлять внутренними, пока они не сведены к:
- конечному списку blocker'ов
- или конечному устойчивому fix plan

Дополнительное жесткое правило:
- reviewer не имеет права публиковать provisional review, если сам еще не завершил `Forward Impact Sweep` по реально затронутым осям
- если после первого сильного finding reviewer собирается продолжать анализ того же diff, это finding должно остаться внутренним до конца полного review-прохода
- согласие пользователя с промежуточным finding, короткое подтверждение или отсутствие возражений не разрешает публиковать второй, третий и последующие review-ответы вместо одного итогового
- если reviewer уже начал публичный review и затем обнаружил новый сильный finding в том же diff, это означает, что первый ответ был преждевременным; в нормальном режиме так происходить не должно
- исключение только одно: reviewer уперся в реальный blocker, который нельзя снять без уточнения, и без этого уточнения дальнейший `Forward Impact Sweep` неизбежно превратится в догадки

Дополнительное операционное правило:
- если reviewer после начала анализа пишет себе что-то вроде `сейчас ещё посмотрю соседние файлы`, `досмотрю ещё одну ось`, `проверю ещё call-site`, это автоматический признак, что review ещё не готов к публикации
- если пользователь повторно вызывает review в той же сессии, reviewer обязан делать новый ответ как полный consolidated review по актуальному diff, а не как продолжение или "вторая итерация" прошлого ответа
- если после локального фикса reviewer подозревает, что могли открыться новые риски на соседних осях, он обязан сначала полностью прокрутить их внутри себя, и только потом отвечать пользователю

### Gate 1: Context Gate

1. Просмотреть diff или измененные файлы.
2. Сопоставить изменения с бизнес-целью, постановкой и `AGENTS.md`, если он есть.
3. Определить режим: `Frontend`, `Backend` или `Mixed`.

### Gate 2: Root Cause Gate

Для каждого сильного finding reviewer обязан:
- сформулировать, какой инвариант или use case реально сломан
- отделить root cause от симптома
- явно отметить, если текущий fix выглядит symptom-driven

Пока root cause не сформулирован, к плану фикса переходить нельзя.

### Gate 3: Candidate Fix Gate

Для каждого finding reviewer обязан мысленно построить минимальный fix, который:
- чинит именно root cause
- не лечит только один path
- не создаёт новый split source of truth
- не меняет поведение за пределами постановки без необходимости

### Gate 3.5: Alternative Fix Check

Для каждого finding уровня `high` или `medium` reviewer обязан быстро сравнить минимум 2 правдоподобных варианта:
- основной candidate fix
- ближайший symptom-fix или другой соблазнительный, но потенциально слабый вариант

Reviewer обязан коротко отвергнуть худший вариант, если он:
- лечит симптом, а не root cause
- создаёт split source of truth
- меняет внешний контракт без необходимости
- добавляет fallback, скрытую деградацию или лишнюю сложность
- вносит I/O или perf-риск без выигрыша в корректности

Не нужно искусственно генерировать много веток. Для `low severity` или `low confidence` findings достаточно одного candidate fix, если сравнение альтернатив не даёт новой информации.

### Gate 4: Forward Impact Sweep

Перед финальным планом reviewer обязан прогнать candidate fix вперёд только по реально затронутым осям и проверить:
- upstream callers
- downstream contracts
- tests и фикстуры
- swagger, examples и docs
- perf/I-O path
- render tree, state и effects, если diff затрагивает UI

На этих осях reviewer обязан проверить:
- не ломает ли fix соседний use case
- не создаёт ли новый split source of truth
- не меняет ли внешний контракт
- не вносит ли новый UI/runtime side effect
- не конфликтует ли fix с архитектурой, бизнес-логикой или `AGENTS.md`

После этого reviewer обязан выполнить дополнительный thought experiment:

`Представь, что fix уже вмержен. Какие новые баги или регрессии теперь наиболее вероятны?`

Оставлять в выводе нужно только те риски, которые подтверждаются кодом, контрактом, UX-flow или архитектурой. Чистые спекуляции и side-effects вне затронутых осей отбрасывать.

Проверка считается незавершенной, пока reviewer не проверил минимум:
- соседние call-site'ы того же измененного API/репозитория/DTO
- write-path’ы и validate-path’ы, если diff меняет read-semantics общего lookup-а
- write-path’ы и validate-path’ы для persisted fields, если предлагается менять domain validation/fallback из-за плохих данных в БД
- read-path’ы, если diff меняет mutation/storage semantics
- публичные endpoint/use case, которые получают измененный DTO или контракт

До прохождения этой проверки reviewer не должен публиковать findings наружу как итог review.

### Gate 5: Final Plan Gate

Если после `Forward Impact Sweep` новых скрытых багов не найдено, reviewer обязан дать конечный план фикса.

Если найден новый риск, reviewer обязан:
- добавить его как finding, если он реальный
- пересобрать candidate fix
- ещё раз пройти `Forward Impact Sweep`

Итог review должен по возможности приводить к устойчивому fix plan, после которого не требуется новый цикл вопросов уровня:
- `как будешь чинить`
- `это точно не symptom fix`
- `что ещё это сломает`

### Gate 6: Publication Gate

Перед публикацией review reviewer обязан пройти короткий self-check. Если хотя бы на один вопрос ответ `нет`, публиковать review нельзя:

1. Проверен ли весь актуальный diff, а не только последний локальный кусок?
2. Проверены ли все реально затронутые пользовательские flow и соседние call-site'ы?
3. Завершён ли `Forward Impact Sweep`, или reviewer всё ещё собирается "досмотреть ещё одну ось"?
4. Является ли этот ответ единым итоговым review по текущему diff, а не продолжением прошлого partial-review?
5. Готов ли reviewer утверждать, что если после отправки найдётся ещё один medium/high finding в том же diff, это будет неожиданностью, а не следствием незавершённого анализа?

Если хотя бы один ответ `нет`, reviewer обязан продолжить внутренний анализ и не писать пользователю findings.

## Core Checks

- Бизнес-смысл: если бы не было падения теста, трейса или лога, требовала бы бизнес-задача именно такого изменения. Если ответ `нет` или `неочевидно`, считать это finding-кандидатом.
- Source of truth: не меняет ли diff источник истины, и не начинает ли новая сущность собираться из нескольких источников без явного бизнес-правила.
- Инвариантная группа: если одно связанное поле стало optional, резолвится из истории или меняет семантику, проверять всю связанную группу полей, а не только один атрибут.
- Downstream contract: если новый код передает данные дальше в handler, service, domain method, calculator или UI flow, проверить, не нарушены ли предусловия следующего слоя.
- Symptom vs root cause: не является ли fix частичным, когда симптом подавлен, а тот же бизнес-инвариант по-прежнему может сломаться соседним путем.
- Bad data vs bad validation: если runtime падает на доменной валидации, проверить, может ли это значение попасть в БД через все текущие entrypoints. Если текущий контракт strict и плохое значение возможно только из старых/битых данных, правильный fix должен быть migration/backfill/data repair; изменение validation/fallback считать finding-кандидатом.
- Alternative fix rejection: для сильных findings проверить, не выглядит ли ближайший альтернативный fix проще только потому, что он маскирует риск или выносит его в другой слой.
- Тестовая ось: есть ли тест именно на рискованную ось сценария, а не просто на happy path рядом.

## Secondary Checks

- Если после fix публичные примеры, swagger examples, error examples или документация продолжают описывать старое ошибочное поведение как нормальное, это считать красным флагом и отдельным finding.
- Reviewer не должен доверять зеленым тестам, если фикстуры по умолчанию маскируют рискованный сценарий. Если дефект зависит от валюты, статуса, роли, временной сортировки, null-ветки или другой чувствительной оси, reviewer обязан проверить, есть ли тест именно на эту ось.
- Reviewer обязан проверять, не появились ли запросы в базу, репозитории, module clients, внешние API или другие I/O-операции внутри циклов. Любой такой паттерн считать finding, если только batching, preloading или иная форма явной оптимизации не невозможны по постановке и это не объяснено прямо в коде или задаче.
- Проверять, не создал ли diff лишнюю сложность, слабую абстракцию, скрытую деградацию, ослабление валидации, мертвую ветку или production-изменение ради тестов, фикстур или локального запуска.

## Frontend-Specific Checks

- `Render Tree Sweep`: reviewer обязан отдельно проверить итоговую JSX-структуру.
- `Existing Pattern Match`: если в проекте уже есть устоявшийся UI-паттерн для статуса, кнопки, секции, иконки, валидации или layout, отклонение без причины считать finding-кандидатом.
- `UI State Integrity`: проверить `saved/dirty/form/query/local` state и derived flags.
- `Effect Stability`: искать циклы `render -> new ref/value -> effect -> setState -> render`.
- `Flow Integrity`: проверить, не усложнил ли diff пользовательский сценарий так, что save/next/edit paths стали неочевидно связаны.
- `Accessibility Surface`: проверить roles, labels, disabled semantics, nested interactive elements, icon-only controls.
- `Invisible Regression`: проверить, не изменило ли “чисто разметочное” изменение defaultOpen, connected state, mounted tree, button availability или save order.

## Anti-Pattern Catalog

- `Symptom Fix`: исправлен симптом, но не восстановлен бизнес-инвариант целиком.
  Как выглядит: обработали только один failing path, но соседний путь ломает тот же сценарий.
  Почему это риск: баг визуально пропал, но система все еще не держит инвариант.
- `Split Source Of Truth`: новая сущность или решение собирается из нескольких исторических источников без явного правила.
  Как выглядит: цену взяли из одной истории, курс, расходы или шаблонные поля из другой.
  Почему это риск: итоговое состояние становится внутренне неконсистентным.
- `Broken Invariant Group`: одно связанное поле обновили, остальные остались в старой семантике.
  Как выглядит: поле стало optional или резолвится из истории, а связанный курс, статус, флаг или расчет остались без синхронного изменения.
  Почему это риск: diff локально выглядит корректно, но ломает связанный набор бизнес-условий.
- `Hidden Fallback`: fallback молча меняет поведение use case или маскирует отсутствие корректных данных.
- `Migration Avoidance`: diff ослабляет validation, добавляет default/fallback или меняет формулу, хотя текущие write-path'ы уже запрещают плохие persisted данные и проблема должна решаться миграцией/backfill/data repair.
- `Template Heuristic`: шаблон, история или приоритетная запись выбираются эвристикой без явного бизнес-правила.
  Как выглядит: код сортирует записи по удобной технической логике, но в задаче не сказано, почему именно эта запись должна победить.
  Почему это риск: деньги, статусы или дедлайны начинают зависеть от молчаливой эвристики.
- `Downstream Contract Violation`: caller начал передавать состояние, которое следующий слой не гарантирует принимать.
- `Green But Blind Tests`: тесты зеленые, но опасная ось сценария не покрыта или замаскирована фикстурами.
  Как выглядит: тест проходит на дефолтных фикстурах, где рискованный флаг, валюта, роль или null-case никогда не активируется.
  Почему это риск: change кажется защищенным тестами, хотя реальный дефект не моделируется.
- `I/O In Loop`: запросы в БД, репозитории, module clients или внешние API происходят внутри цикла без явной причины.
  Как выглядит: внутри `foreach` вызывается repository, database query, module client или внешний API.
  Почему это риск: появляется N+1, нестабильная latency и деградация под ростом входных данных.
- `Public Contract Drift`: Swagger, examples или docs закрепляют уже неверную или устаревшую семантику.
- `Test-Driven Runtime Change`: production-код изменен в первую очередь ради удобства теста, фикстуры или локального запуска.
- `Dirty Render Tree`: в JSX остались старые и новые ветки одновременно, дублируются секции, формы, props или actions.
- `Pattern Drift`: diff вводит новый визуальный или структурный паттерн там, где рядом уже есть устоявшийся.
- `Split UI State`: одна пользовательская сущность собирается из нескольких независимых UI-состояний без явного правила.
- `Unstable Effect Loop`: derived value создаётся заново на каждом render и триггерит effect/setState цикл.
- `Flow Coupling`: визуально отдельный блок реально сохраняется только через соседнюю кнопку или другой submit path.
- `Invisible Regression`: изменение “только разметки” реально меняет connected state, defaultOpen, mounted tree, button availability или save order.

## Приоритизация Findings

- Сначала поднимать проблемы, которые ломают бизнес-семантику, инварианты, контракты или создают явную регрессию runtime-поведения.
- Сортировать findings сначала по severity, затем по confidence. Низкоуверенные подозрения не должны вытеснять явные баги из начала ответа.
- Если в diff уже есть 1-3 сильных findings, не нужно искусственно раздувать ответ слабыми замечаниями ради полноты. Но это не жесткий лимит: добавляй дополнительные findings, если они независимы, дешевы в объяснении и реально повышают ценность review.
- Слабые замечания включать только если они не размывают основные проблемы и не мешают пользователю быстро понять, что именно нужно исправить в первую очередь.
- Если findings нет, написать это явно и кратко перечислить остаточные риски или пробелы в проверке.

## Блокирующие вопросы

- Если reviewer не может надежно определить бизнес-семантику изменения без догадок, нужно явно остановиться и сформулировать блокирующий вопрос вместо продолжения ревью на предположениях.
- Если в diff есть несколько правдоподобных трактовок source of truth, выбора исторической записи, fallback-логики, правил приоритизации или границ use case, и из кода это не выводится однозначно, это блокирующий вопрос.
- Если непонятно, является ли поведение багом, сознательным компромиссом или частью внешнего контракта, reviewer должен сначала отметить эту неопределенность, а не выдумывать intent автора.
- Если для качественного вывода не хватает постановки, связанного diff, исходного лога, примера запроса, схемы данных или соседнего контракта, reviewer должен коротко сказать, чего именно не хватает.
- Блокирующий вопрос не отменяет уже найденные явные findings. Их нужно перечислить отдельно, а вопрос использовать только для частей diff, где без уточнения неизбежны догадки.

## Формат ответа

Использовать ровно один из двух режимов вывода.

### Режим 1: обычный review-текст

Использовать по умолчанию.

Правила:
- писать по-русски, если пользователь явно не попросил другой язык
- начинать с findings, отсортированных по серьезности
- давать ссылки на конкретные файлы и строки, когда это возможно
- после findings можно добавить короткий блок `План фикса`, если gates пройдены и reviewer может предложить устойчивое решение
- в `Плане фикса` указывать:
  - что менять
  - какие слои/файлы затронуть
  - какие риски fix сознательно не должен задеть
  - почему ближайший альтернативный fix отвергнут, если `Alternative Fix Check` дал нетривиальный результат
  - какими тестами/сценариями подтверждать fix
- если findings нет, написать это явно и кратко перечислить остаточные риски или пробелы в проверке

### Режим 2: inline findings через `::code-comment`

Использовать только когда:
- пользователь явно просит inline comments
- среда поддерживает review-директивы, и есть смысл привязать finding к конкретному месту

Правила:
- `title` и `body` тоже писать по-русски, если пользователь явно не попросил другой язык
- один finding = одна директива
- каждая директива должна быть на отдельной строке
- не оборачивать директивы в Markdown-списки
- не оборачивать директивы в fenced code block
- не смешивать директивы с bullet-list findings выше них
- после блока директив можно добавить только короткий обычный русскоязычный блок `План фикса`, если gates пройдены
- если findings нет, не эмитить директивы вообще; вместо этого написать обычный короткий текст

## Жесткие требования к качеству вывода

- Не смешивать сломанный Markdown, полураскрытые code fences и директивы.
- Не писать findings одновременно и bullet-списком, и `::code-comment` директивами.
- Не расплываться в общих архитектурных рассуждениях без evidence из diff.
- Не предлагать большой рефакторинг, если текущий diff сам не создал для него необходимости.
- Считать несоответствие бизнес-задаче проблемой первого класса, даже если код технически работает.
- Явно отмечать, когда symptom-driven change ослабляет поведение системы вместо исправления корневой причины.
- Перед финальным выводом обязательно пройти `Root Cause Gate` и `Forward Impact Sweep`.
- Если reviewer может предложить устойчивый fix plan, он обязан сделать это в том же ответе, не дожидаясь отдельного вопроса пользователя.
- Не выдавать несколько последовательных review-ответов по одному и тому же diff только потому, что новые findings были найдены уже после первого ответа. В нормальном режиме все сильные findings должны быть собраны в один итоговый review после полного sweep.
